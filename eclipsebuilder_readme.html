Description

This document describes how to run a build from the command line which produces all Eclipse SDK and runtime products. The PDE-BUILD, Ant and the JDT Core Eclipse components are the techologies from build 2.1 M3 are used in this process. These, along with scripts that drive the process, are contained in the combined resources of org.eclipse.releng.basebuilder and org.eclipse.releng.eclipsebuilder, two projects available from the dev.eclipse.org Eclipse repository.

PDE-BUILD is a technology in Eclipse which generates build scripts for features, plugins and fragments. The script generation functions are made available via custom Ant tasks (see documentation on PDE-BUILD). Calls to these tasks are contained in static Ant scripts provided in the projects noted above. The JDT Core compiler is used for compiling the java source code in Ant javac calls. The Ant technology in Eclipse is used to run both static and generated Ant scripts.

This is the mechanism used by Eclipse Release Engineering.

 

System Requirements

The Eclipse plugins and fragments for this mechanism were taken from Windows and Linux-Motif installations of the Eclipse 2.1 M3 build. It is recommended that the builds are conducted on one of these two platforms.

The following will also be required:

  1. Sun or IBM JDK 1.3 or higher
  2. (On Linux only) Windows version of the rt.jar (bootstrap classes, classes that comprise the Java platform), version 1.3.* or higher
  3. CVS 1.10 or higher
  4. Info-ZIP Zip version 2.3 or higher
  5. Info-ZIP UnZip version 5.41 or higher
  6. GNU tar version 1.12 or higher
  7. >1 Gigabyte free disk space (23 zip files, 6-67 meg in size, scripts, all Eclipse feature, plugin and fragment projects from dev.eclipse.org)
  8. org.eclipse.releng.basebuilder
  9. org.eclipse.releng.eclipsebuilder

 

Initial Setup

  1. Install the JDK, CVS client, Info-ZIP Zip and Unzip, and GNU tar as per each program's instructions.
  2. Add the JDK/bin directory, zip, unzip and tar executables to your path.
  3. Set the JAVA_HOME environment variable to the JDK installation directory.
  4. (Windows 98) Increase the default environment space as specified here.
  5. Export the projects org.eclipse.releng.basebuilder and org.eclipse.releng.eclipsebuilder to an empty directory created near the root of a drive to keep path lengths to a minimum. The build may fail due to some paths used in file copy operations exceeding 255 characters.

    cvs -d :pserver:anonymous@dev.eclipse.org:/home/eclipse export -r <tag> -d <builderDirectory> org.eclipse.releng.basebuilder

    cvs -d :pserver:anonymous@dev.eclipse.org:/home/eclipse export -r <tag> -d <builderDirectory> org.eclipse.releng.eclipsebuilder

    The files from both projects must be in the same directory for path calculations.

  6. (Linux) set execute permission on the buildEclipse shell script, "chmod 755 buildEclipse".

     

Run the Build

The scripts buildEclipse.bat and buildEclipse are included in the org.eclipse.releng.eclipsebuilder project to invoke the Eclipse AntRunner on the Ant build scripts.

To execute, use the following command from the directory containing the build scripts specifying any of the optional parameters described below:

buildEclipse [buildid name][buildLabel name][-buildType I|N|S|R|M][-bootclasspath path][-mail] [-mapVersionTag tag][postingDirectory path]

 

Parameter Description
buildid Default set to<buildType><datestamp>, i.e. I20021217, but can be set to any String value.
buildLabel The name of the directory that will contain all the products of the build. Default set to <buildType>-<buildid>-<timestamp>, ie. I-I20021217-200212170800. This particular nomenclature is used internally for php scripts web, but can be set to any String value.
buildType

The buildType specification is used to identify the build as one of the following:

I = Integration
N= Nightly
M= Maintenance
R= Release
S= Stable
T= Test

Specifying 'N' as the buildType will retrieve the HEAD stream of all plugins, fragments and features listed in the *.map files instead of using the tags specified. Typically, I and N are used for building the latest development stream of Eclipse. M can be used for building a maintenance stream of Eclipse. R and S are typically not used since these are usually the result of promoted Integration builds.

bootclasspath Absolute path to the classes that implement the Java 2 Platform, also known as bootstrap classes. These are typically contained in an rt.jar, inluded with a JRE or JDK. The default value set to the location of the rt.jar included with the installed JDK. If building on Linux, the bootclasspath should be set to a Windows version of the rt.jar. The Windows version allows all os specific plugins and fragments to compile, whereas the Unix versions will not allow the Windows specific plugins and fragments to compile cleanly.
mail

Set this flag if you wish to send mail when a build starts. Setting this flag will require a monitor.properties file created in the same directory containing the exported build scripts. At a minimum, the monitor.properties file will require key value entries such as the following:

host=aserver.ca.ibm.com
sender=myemailaddress
recipients=address1,address2,address3
mapVersionTag The tag of the org.eclipse.releng project to checkout from dev.eclipse.org. The default value is set to HEAD. Use a specific tag to build an integration build previously posted on downloads.eclipse.org. The tag names match the date and time of the desired build with a 'v' prefix, i.e. v200203071430.
postingDirectory The directory to which the products of the build should be copied. The product of the build is a directory, named with the buildLabel (see description above), containing the zip files of the SDK and runtime products and web pages. Default value set to current directory.

 

Known Limitations

  1. This mechanism includes linux-motif and windows installations of Eclipse only. These scripts have only been tested on Linux 7.2 and Windows 2000\XP.
  2. Building unix flavours of Eclipse on Windows will result in incorrect file permissions. The *.so, *.sl, and Eclipse executable files will not have execute permissions. In addition, linux-motif builds will not have symbolic links (libXm.so and libXm.so.2) to libXm.so.2.1 in the installation root directory. In these cases you would have to write an install or post install script to set permissionss properly. (UNIX builds built on linux-motif will set sufficient permissions and make the appropriate soft links).
  3. When the Mac port of Eclipse is built on a Windows operating system, a soft link to the root of the eclipse installation directory will be missing. The following post-installation steps will be required:
    • cd to Eclipse.app/Contents/Resources
    • create a symbolic link to the eclipse installation directory by executing the following command: "ln -s Java ../../.."
    • cd to ../Contents/MacOS
    • set execute permission on the eclipse file by executing this command: "chmod 555 eclipse"
  4. Only Java source code is compiled. *.dll, *so, *sl and Eclipse executables are all checked out of dev.eclipse.org prebuilt in this build process.
  5. This process builds all Eclipse runtime and SDK products requiring > 1 gig of hard disk space on the build machine.
  6. InfoZIP Zip and Unzip are used to preserve file permissions when extracting or creating zip files. The dependency on using these should be removed in future releases by using the Ant tar and zip/unzip tasks instead.
